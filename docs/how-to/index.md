# How-to guides

How-to guides are the *application + action* quadrant: focused
recipes for the already-competent user who knows what they want to
do. Each guide solves one problem; titles state the task precisely;
steps are executable.

If you are learning the project for the first time, start at the
[Tutorials](../tutorials/index.md) instead. For surface catalogues,
see the [Reference](../reference/index.md). For the *why* behind the
design, see the [Explanation](../explanation/index.md).

## Discipline

A how-to guide earns its place here when it:

- Has a *task-shaped* title — "Install X", "Configure Y", "Migrate
  from Z" — not a noun-shaped title (those are reference) or a
  question-shaped title (those tend to be tutorials or explanations).
- Assumes the reader knows what they want and will choose. Where a
  tutorial holds the reader's hand, a how-to gives them options.
- Stays scoped to one problem. If a guide grows past three sub-tasks,
  it is probably two guides.
- Lists prerequisites at the top and the success check at the bottom.

If a page reads like a story ("first we'll set up, then we'll see…"),
it has drifted into [tutorials](../tutorials/index.md). If a page
reads like a catalogue ("the X command takes flags A, B, C…"), it has
drifted into [reference](../reference/index.md).
